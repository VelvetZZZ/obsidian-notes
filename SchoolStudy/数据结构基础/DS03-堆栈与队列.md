### 1-1 Run the following operations on a stack S: Push(S,1), Push(S,2), Pop(S), Push(S,3), Pop(S), Pop(S). The output sequence must be {1, 2, 3}. <font color="#ff0000">F</font>

##### <span style="background:rgba(236, 57, 57, 0.55)">栈：</span>
什么是栈？
- 栈（Stack）：是一种数据结构。
- 核心原则：LIFO (Last-In,First-Out)，即“<font color="#c00000">后进先出</font>”。
- 记忆比喻：薯片桶子里最后放进去的那片薯片（Push），一定是最先拿出来吃掉的（Pop）。
这道题的实际输出顺序是 `2, 3, 1`。
### 1-2 If keys are pushed onto a stack in the order {1, 2, 3, 4, 5}, then it is impossible to obtain the output sequence {3, 4, 1, 2, 5}.  <font color="#ff0000">T</font>

#####  为什么 `3, 4` 是 _可能_ 的？

我们来模拟一下“薯片筒” (Stack)：

1. **想拿到 `3`：**
    
    - `Push(1)` -> 栈：`[1]`
        
    - `Push(2)` -> 栈：`[1, 2]`
        
    - `Push(3)` -> 栈：`[1, 2, 3]` (3 在最上面)
        
    - `Pop()` -> **输出 3**。
        
    - **此时栈：`[1, 2]`** (2 在最上面)
        
2. **想拿到 `4`：**
    
    - `Push(4)` -> 栈：`[1, 2, 4]` (4 在最上面)
        
    - `Pop()` -> **输出 4**。
        
    - **此时栈：`[1, 2]`** (2 在最上面)
        

- **结论：** `...3, 4...` 这个开头是**完全可以做到**的！
    

---

######  为什么 `...1, 2...` 是 _不可能_ 的？

- 我们接着上面的模拟。现在输出是 `3, 4`，栈里是 `[1, 2]` (2 在 1 上面)。
    
- **目标：** 接下来输出 `1`。
    
- **现实：** 栈顶是 `2`。你 `Pop()` 一下，**只能**拿到 `2`！
    
- 你**永远不可能**在 `2` 还没被拿走（Pop）的时候，就“隔空取物”把底下的 `1` 先拿走。

###### “栈的铁律”

- 在入栈顺序 `...1, 2, ...` 中，`2` 是在 `1` **之后**入栈的。
    
- 这意味着 `2` **一定**在 `1` 的**上面**（或者 `1` 已经出栈了）。
    
- 只要 `1` 和 `2` 都在栈里，`2` **必须**比 `1` **先** Pop 出来。
    
- 而这个题目要求的输出顺序是 `...1, 2...`，它要求 `1` **先于** `2` 被 Pop 出来。
    
- 这**彻底违反**了“后进先出” (LIFO) 的原则。

##### **总结：** 这道题是 **T (True)**，它确实是“impossible”的。 但“罪魁祸首”不是 `3, 4`，而是 `1, 2`。<span style="background:rgba(136, 49, 204, 0.2)">在 `1, 2` 这个子序列中，2 必须在 1 之前出栈</span>，但输出要求 1 在 2 之前出栈。

### 1-4 Given that the pushing sequence of a stack is { 1, 2, ⋯, n } and popping sequence is { x1​,x2​,⋯,xn​ }. If x2​=n, we can obtain n−1 different possible popping sequences. <font color="#ff0000">T</font>

翻译：<font color="#e6b7d5">已知一个栈的入栈顺序为 {1, 2, ..., n}，出栈顺序为 {x₁, x₂, ..., xₙ}。如果 $x_2 = n$（第二个出栈的元素是n），那么我们可以得到 n-1 种不同的可能出栈序列。</font>✅
##### 解析
- 每一个 `x_1` 的选择（`x_1=1` 或 `x_1=2` ... 或 `x_1=n-1`），都**唯一对应**了一个合法的、满足 `x_2=n` 的出栈序列。
- `x_1` 有 **`n-1`** 种选择。
- 因此，总共有 **`n-1`** 种不同的可能出栈序列。

##### <font color="#d83931">递归</font>程序几乎总是比等效的非递归（<font color="#fac08f">迭代</font>）程序**更慢**，并且更占用空间：

- **迭代 (Iteration) 的优势：** “非递归”程序（即 `for` 或 `while` 循环）没有这个开销。它的“管理工作”仅仅是把一个计数器 `i` 加 1，这个操作比函数调用**快上千倍**。

原题把两者的优缺点**完全搞反了**。

##### <font color="#e36c09">顺序栈(Sequential Stack)：</font><font color="#e36c09"><span style="background:rgba(240, 107, 5, 0.2)"><font color="#e36c09">用数组来实现的栈</font></span></font>
- **比喻 (造“薯片筒”)：**
    - 你在造“薯片筒”之前，**必须**先跟“系统”**预订一个“固定大小的盒子”**。
    - 比如你写：`int my_stack[100];`
        
    - 这就是说：“我这个‘薯片筒’，**最多、永远**只能装 100 片薯片。我**提前**把这 100 个位置都占好了。”
        
- **核心问题 (缺点)：**
    - **“栈满” (Stack Full)：** 当你往这个“盒子”里 `Push` 了第 100 片薯片后，你再想 `Push` **第 101 片**...
    - **BOOM！** 盒子满了！程序崩溃！
    - 这就是“会出现<font color="#d83931">栈满</font>的情况”。你**必须**在写代码时**猜**一个“最大值”。

##### <font color="#d83931"> 链栈 (Linked Stack)：</font><font color="#d83931"><span style="background:rgba(236, 57, 57, 0.55)"><font color="#d83931">用链表来实现的栈</font></span></font>
- **比喻 (造“薯片筒”)：**
    - 你**不预订**任何“固定大小”的盒子。
    - **“栈”**这个概念，就是你手里的 `top` 指针（`Head` 乘务员）。
        
    - **`Push(S, 1)` (放第1片)：**
        
        - 你跟“系统”说：“请给我**一块**新内存（一个新车厢）。” (`malloc`)
            
        - 你把 `1` 放进去。`top` 指向它。
            
    - **`Push(S, 2)` (放第2片)：**
        
        - 你**再次**跟“系统”说：“请再给我**一块**新内存。”
            
        - 你把 `2` 放进去，让它的 `next` 指向 `1`。`top` **移动**到 `2` 上面。
            
    - **`Push(S, 3)` (放第3片)：**
        
        - 你**又**要了一块新内存...
            
- **核心优势 (优点)：**
    
    - **“栈满” (Stack Full)：** 什么时候会“满”？
        
    - **答案：** 你的“盒子” (链表) 是**动态增长**的！
        
    - 它**永远不会**因为你“设计”的 `100` 个限制而满。
        
    - 它“**通常不会**出现栈满”。
        
    - **（唯一的“满”：** 当你把整个电脑的 16GB 内存**全部**用完了，`malloc` 再也拿不到新内存了，那才会“满”。但这已经是“系统崩溃”，而不是你“栈”的设计问题了。)

##### <font color="#d83931">队列 (Queue)</font>
**“队列” (Queue)** 是一种**抽象数据类型 (ADT)**。
- **核心原则：** 它的唯一法则是 <span style="background:rgba(236, 57, 57, 0.55)">“先进先出” (First-In, First-Out, <b>FIFO</b>)</span>**。
    
- **操作限制：** 它是一种**受限制**的“线性表”（和“栈”一样）。
    
- **专属术语：** 这种限制体现在它必须在**两个不同的端点**进行操作：
    
    1. **队尾 (Rear)：** 只允许**插入**新元素（称为 **`Enqueue` / 入队**）。
        
    2. **队首 (Front)：** 只允许**删除**旧元素（称为 **`Dequeue` / 出队**）。

**总结 (蓝图)：** “队列”是一份“设计蓝图”，它**只规定了“必须先进先出”这个规则**。它**不关心**你是用“数组”还是“链表”去实现它。

##### <font color="#c766e9">“循环队列” (Circular Queue)</font> 
**不是**一种抽象数据类型，它是一种**具体的“实现” (Implementation)**。

<span style="background:rgba(240, 200, 0, 0.2)">定义：循环队列（**Circular Queue**）是一种特殊的队列结构，用**顺序存储（数组）实现，但它把存储空间的首尾首尾相连**，形成一个“环形”的逻辑结构。</span>
- **实现材料：** “循环队列”这个术语**专指**——使用 **“数组” (Array)** 作为“建筑材料”去**实现**“队列”蓝图。
    
- **解决的问题：** 它的存在是为了解决“普通顺序队列（数组队列）”的<span style="background:rgba(136, 49, 204, 0.2)">“假溢出” (False Overflow)</span>  缺陷。
    
    - _（“假溢出”：指数组尾部已满 (`Rear` 撞墙)，但数组头部因为 `Dequeue` 操作已经空出空间，却无法被 `Enqueue` 利用的“空间浪费”现象。）_
        
- **核心机制：**
    
    1. 它在**逻辑上**把一个“线性”的数组**“掰弯”**成了一个**“环形”**。
        
    2. 它在**技术上**通过 **“取模运算” (`%`)** 来实现这个“环”。
        
    3. 当 `Rear` 或 `Front` 索引“走”到数组的物理末尾时，`%` 运算会自动让它**“回卷” (Wrap-around)** 到索引 `0`，从而**重复利用**数组头部的空闲空间。

**总结 (具体实现)：** “循环队列”是一种**使用“数组”和“取模运算”** 来高效实现“队列”蓝图的**特定技巧**。这个术语**专属于数组**，与“链表”无关。

# 我的疑问：

### 一、数组、链表和栈

> **“数组和链表为什么都能和栈结合啊，栈不是和它们并列的数据结构吗？”**

**“开悟”时间到。** **栈 (Stack)** 和 **数组 (Array) / 链表 (Linked List)** **不是并列关系**！

它们是**“蓝图”**和**“建筑材料”**的关系。

#### A. 抽象数据类型 (ADT)：“栈” (蓝图)

- “栈” (Stack) 是一个**“抽象”**的概念，它是一份**“设计蓝图”**。
    
- 这份“蓝图”只规定了“**规则**”和“**功能**”：
    
    1. **规则 (LIFO)：** 必须“后进先出”。
        
    2. **功能 (接口)：** 必须提供 `Push()` (插入) 和 `Pop()` (删除) 这两个“按钮”。
        
- **蓝图“不关心”你用什么材料：** “蓝图”**才不管**你是用“木头”还是“钢铁”来造这栋楼。
    

#### B. 具体实现 (Implementation)：“数组”和“链表” (建筑材料)

- “数组” (Array) 和 “链表” (Linked List) 是**“具体”**的、**“物理”**的结构。
    
- 它们是两种不同的**“建筑材料”**，规定了数据**在内存中“如何存放”**：
    
    1. **数组 (Array)：** 像“电影院连排座位”，所有数据**连续**存放。
        
    2. **链表 (Linked List)：** 像“寻宝游戏”，数据**分散**存放，靠“指针”串联。
        

**结论：**

- 你是“建筑师”（程序员）。
    
- 你拿着“栈”这份“**蓝图**”(ADT)。
    
- 你**必须选择一种“材料”** (Implementation) 去把它**造出来**。
    
    - 你选择用“**数组**”当材料 -> 你造出来的叫 **“顺序栈”** (Sequential Stack)。
        
    - 你选择用“**链表**”当材料 -> 你造出来的叫 **“链栈”** (Linked Stack)。
        

所以，它们**不是并列的**，而是**“抽象”**和**“具体”**的**上下层关系**。

### 3. 两种“栈”的实现举例

现在，我们再看“顺序栈”和“链栈”，你就彻底懂了（我们用 `Push(5)` 和 `Pop()` 来举例）：

#### 举例 1：顺序栈 (用“数组”实现)

- **材料：** 一个**固定大小**的数组 `data[100]` + 一个**索引** `top`。
    
- **初始化：** `top = -1` (表示栈是空的)。
    
- **`Push(5)` (插入)：**
    
    1. `top` 增加 1 ( `top` 变成 `0` )。
        
    2. 把 `5` 放入 `data[top]` (即 `data[0] = 5` )。
        
- **`Pop()` (删除)：**
    
    1. 从 `data[top]` (即 `data[0]` ) 取出 `5`。
        
    2. `top` 减少 1 ( `top` 变回 `-1` )。
        
- **(这就是 1-7 题的知识)：** 它的**缺点**是数组 `data[100]` 大小固定，`Push` 到第 101 个时就会 **“栈满”**。
    

#### 举例 2：链栈 (用“链表”实现)

- **材料：** 只有一个**指针** `top` (它指向“栈顶”的那个节点)。
    
- **初始化：** `top = NULL` (表示栈是空的)。
    
- **`Push(5)` (插入)：**
    
    - (这就是你学过的“**在链表头部插入**”，`p->next=s` 的“反向”)
        
    
    1. 创建 1 个**新节点** (new node)，把 `5` 放进去。
        
    2. 让这个新节点的 `next` 挂钩**指向** `top` 所指的“旧栈顶” ( `new->next = top` )。
        
    3. 把 `top` 指针**移动**到这个“新节点”上 ( `top = new_node` )。
        
- **`Pop()` (删除)：**
    
    - (这就是你学过的“**删除链表头部**”，`p->next=s->next` 的“变体”)
        
    
    1. 从 `top` 指向的节点中取出 `5`。
        
    2. **记住** `top` 的“下一个”节点 ( `temp = top->next` )。
        
    3. **销毁** `top` 指向的“旧栈顶” ( `free(top)` )。
        
    4. 把 `top` 指针**移动**到 `temp` (即 `top = temp` )。
        
- **(这就是 1-7 题的知识)：** 它的**优点**是**没有**固定大小，只要内存够，就能一直 `Push`，**“通常不会栈满”**。
